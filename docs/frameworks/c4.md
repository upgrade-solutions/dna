# DNA ↔ C4 Model

How DNA's three layers map to the **C4 Model** (Simon Brown, 2018). C4 is a software-architecture diagramming technique with four levels of abstraction (Context → Container → Component → Code). Where DNA describes the *business* and lets code generation handle the rest, C4 starts with software and zooms in. They overlap most clearly at the System Context and Container levels.

> **Note (post-reorg, 2026-04):** This doc was written before the People/Entities/Activities reorg. Some examples reference the old "Resources-used-as-Roles" structural-typing model. The current shape splits Person, Role, Group, and Membership into first-class primitives — see [`README.md`](../../README.md) and [`ROADMAP.md`](../../ROADMAP.md). The conceptual mappings below still hold; concrete syntax examples may be outdated.

> Of the frameworks in this folder, C4 has the lightest Operational-layer overlap. It's strongest at the boundary between DNA's Product and Technical layers — useful for cross-team architecture communication, less useful for capturing business logic.

## Level-by-level mapping

C4's four levels each map to a different DNA layer (or are out of scope):

| C4 level | C4 scope | DNA equivalent |
|---|---|---|
| **Level 1: System Context** | The system and its external dependencies (people, other systems) | DNA `Domain` (the named bounded context) + Operational `Resource`s representing external systems + cross-domain `Signal`s consumed/published. Actors map to Resources used as Roles. |
| **Level 2: Container** | The deployable units inside the system (apps, services, databases) | DNA `Cell` (Technical layer) — a Cell is the unit of deployment. Containers in C4 ≈ Cells in DNA, with `Construct`s providing the underlying compute/storage primitives. |
| **Level 3: Component** | The major building blocks inside a Container | Not formally modeled in DNA. Components emerge from how Cells consume DNA — an api-cell generated from `product.api.json` produces components per `Endpoint`/`Operation`. The grouping isn't first-class. |
| **Level 4: Code** | Classes/functions inside a Component | Out of scope for DNA. Generated code; inspect the per-cell output. |
| **Supplementary diagrams** (Deployment, Dynamic, Landscape) | | Deployment ≈ Technical layer's `Environment` + `Provider`. Dynamic diagrams ≈ DNA `Process` flowcharts (see `output-mermaid`). Landscape diagrams ≈ multiple Domains side-by-side; DNA doesn't formally model landscape but the Domain hierarchy serves the same role. |

## Concept-by-concept (C4 element types)

| C4 element | DNA equivalent | Notes |
|---|---|---|
| **Person** (user role) | `Resource` referenced as actor (Role) | Same as Yellow stickies in [Event Storming](./event-storming.md). C4's "Person" is DNA's Resource-used-as-Role — IT support, Customer, Admin, etc. |
| **Software System** (the focus system) | `Domain` (root or sub-domain) + the DNA documents under it | The "system" in C4 corresponds to a DNA Domain hierarchy and its operational/product/technical files. |
| **External Software System** | A `Resource` representing the integration + a separate `Domain` for the integration's surface | DNA's pattern for external systems is either an integration-* package (consuming/publishing through Signals) or a separate bounded `Domain` (e.g., `acme.integrations.salesforce`). |
| **Container** (web app, mobile app, database, message bus) | DNA Technical `Cell` | A C4 Container is "a runtime thing that hosts code" — that's exactly DNA's Cell concept. Cells consume specific DNA documents (an api-cell consumes product.api.json) and generate concrete artifacts. |
| **Component** | Implicit — not formally modeled | Components emerge from the Cell's output; they're not authored in DNA. |
| **Code element** | Out of scope | DNA stops at the Cell boundary; code is generated. |
| **Relationship / arrow** | DNA `Relationship` (between Resources); `Signal` consumption (between Domains/Cells); `Trigger.source: webhook` (inbound from external) | C4 arrows are typed in two ways: dependency direction and protocol. DNA splits these — Resource-level Relationships for data; Signals + Triggers for behavioral coupling. |

## Where DNA intentionally differs

1. **DNA describes business, C4 describes software.** A C4 System Context says "we have a Banking System that talks to Stripe and SendGrid"; a DNA document says "Loans get Approved by Underwriters within BankDepartments and emit lending.Loan.Disbursed Signals." Different starting points.
2. **DNA omits Components by design.** The Component level in C4 is fluid (one team's "service" is another's "module"). DNA doesn't take a position — what gets generated as a component depends on the cell, not the description.
3. **DNA's Container ≈ Cell, but Cells are richer.** A Cell declares which DNA layer it consumes (`product/api`, `product/ui`, `operational/<resource>`), which adapter implements it, which Constructs back it, and which Variables/Outputs flow into/out of it. C4 Containers are diagrammatic; Cells are executable specifications.
4. **C4 has Deployment + Dynamic + Landscape diagrams as separate views.** DNA expresses these from the same source: Deployment from `Environment` + `Provider`; Dynamic from `Process` (rendered via `output-mermaid`); Landscape from the Domain hierarchy.

## Concrete translation example

A C4 System Context for a Lending platform:

- **Person**: Borrower, Underwriter, LendingManager
- **Software System (focus)**: Lending Platform
- **External Software System**: Credit Bureau API, ACH Network, Email Provider
- **Relationships**: Borrower submits applications via the Lending Platform; Lending Platform calls the Credit Bureau API for scoring; Lending Platform settles via ACH Network; Lending Platform sends notifications via Email Provider.

A C4 Container Diagram for the same:

- **Containers**: Web App, Mobile App, API Service, Loan Database, Worker (for nightly delinquency sweep), Notification Bus

**DNA equivalent**: see [`examples/lending/operational.json`](../../examples/lending/operational.json) plus the (not-yet-written) `examples/lending/technical.json`. Concretely:

- Person → `Resource` (Borrower, Underwriter, LendingManager — used as Roles)
- Software System (focus) → `Domain` (`acme.finance.lending`) + the DNA docs under it
- External Software System → `Resource` for each external system + `Trigger` (`source: webhook`) for inbound + `Outcome.emits` for outbound
- Containers (Web App, API Service, Worker, Database) → DNA Technical `Cell` per container, each declaring its `dna` source (`product/ui`, `product/api`, `operational/loan`, etc.) and the `Construct`s it depends on
- Notification Bus → an event-bus-cell that publishes `lending.Loan.Disbursed` Signals to subscribers

The Web/Mobile vs API split — the typical Container-diagram thing — is captured by which `Cell` consumes which `dna` source. Two cells (`web` and `api`) consuming `product/ui` and `product/api` respectively is the DNA equivalent of two C4 containers communicating.

## Where C4 is genuinely better

- **Quick architecture communication.** A C4 diagram on a wiki gets across the shape of a system in 30 seconds; a DNA document doesn't (and isn't trying to). For onboarding, exec briefings, and cross-team sync, render a C4 diagram from your DNA — don't expect readers to parse the JSON.
- **Multi-system landscape.** C4's Landscape diagram (multiple systems side-by-side) is more practiced than DNA's "many domains side-by-side" pattern.
- **Decomposition discussions.** Where teams genuinely debate "is this one component or three" or "should this be its own container," C4 has the vocabulary. DNA stays out of these debates by leaving Components implicit.

## Recommended workflow

1. **Sketch C4 Context + Container diagrams first** during architecture exploration.
2. **Convert once stable**: Personas/External Systems become Resources; Containers become Cells; Containers' adapters become DNA Technical adapters.
3. **Author DNA** as the source of truth going forward; render C4-style diagrams from it for human consumption (an `output-c4` adapter is a natural addition — not yet shipped).
4. **Keep C4 around for the conversations DNA isn't optimized for**: high-altitude cross-system landscape, exec communication, refactoring debates.
